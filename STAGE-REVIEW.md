## S1: ツリートポロジー確定

**現状（VimQuest.md セクション2.4）**:  
大まかなツリー構造は示されているが、全ノード・全接続は未確定。

```
現状の記載:
[h j k l] → [x] → [w b e] → [dw / de / db]
                    └→ [0 $] → [d$ / d0]
                               └→ [^]
```

**議論すべき点**:
1. ツリーの全ノード定義（1ノード＝1〜複数コマンドのセット）
2. ノード間の依存関係（前提→後続の矢印）
3. 分岐ポイントと合流ポイント
4. ツリーの深さ（最長パスのノード数）
5. ツリーの幅（同時にavailableになるノード数の最大値）
6. 総ノード数

**制約（確定済み）**:
- v1.0 コマンド数: 約74個（Op8 + Motion30 + TextObj16 + Instant20）
- 移動と編集を交互に導入する原則
- テキストオブジェクトが中盤のハブとなる
- ドットリピート（`.`）が感動の最高峰

**議論**: （未実施）

## S4: ライフ / ☆閾値の設計公式

**現状（VimQuest.md セクション2.1）**:
- ☆3解法 → ☆2解法 → ☆1解法をまず設計
- ☆3閾値 = ☆3解法ダメージ
- ☆2閾値 = ☆2解法ダメージ  
- ☆1閾値 = ☆1解法ダメージ
- ライフ = ☆1解法 + 余裕2-3

**議論すべき点**:
1. ☆3/☆2/☆1の想定解法をどう設計するか（☆3=最適解、☆2=次善、☆1=冗長だがクリア可能）
2. ☆間のダメージ差の目安（☆3と☆2の差が2-3？ ☆2と☆1の差が3-5？）
3. ライフの余裕幅の基準（固定値？ 比率？）
4. チュートリアルステージは☆閾値を緩くする？
5. Insert入力を含むステージでの計算上の注意点
