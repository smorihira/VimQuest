## 2. ステージ種別

### 2.1 種別定義

| 種別          | 記号 | 目的               | 変更箇所        | テキスト行数                     | ☆3解法(opt) |
| ------------- | ---- | ------------------ | --------------- | -------------------------------- | ----------- |
| **Teach**     | T    | 新コマンド導入     | 1箇所           | 3-5行（序盤）/ 5-8行（中盤以降） | 2-4         |
| **Practice**  | P    | 定着・パターン認識 | 2-3箇所（同種） | 5-10行                           | 4-8         |
| **Challenge** | C    | 複合応用・パズル   | 3-5箇所（混合） | 8-20行                           | 6-15        |

### 2.2 手札方針

| 種別      | 手札構成                                        |
| --------- | ----------------------------------------------- |
| Teach     | 新コマンド + 必要最小限の既習得（解法を明確に） |
| Practice  | 新コマンド + 同ノード全コマンド + 主要既習得    |
| Challenge | 習得済み全コマンド                              |

#### 常時利用可能コマンド

`u`、`Ctrl+R`、`Esc` は全ステージで常に使用可能（`ALWAYS_ALLOWED`）。これらは `availableCommands` に含めず、カード表示もしない。プレイヤーはいつでもundo/redo/Escが使える。

#### Visual専用コマンド（visualCommands）

ステージ定義に `visualCommands?: string[]` を指定すると、そのコマンドは **Visualモード中のみ** 手札に追加される。NORMALモードでは表示も使用もされない。

```typescript
{
  availableCommands: ['h', 'j', 'k', 'l', 'v', 'V'],
  visualCommands: ['d'],  // Visualモード時のみ手札に追加
}
```

用途: `d` をオペレータ（`dw` 等）としてではなく、Visual選択削除専用として提供したいステージ（N37, N38等）。

### 2.3 外科的基準

変更行 ≤ テキスト全体の50%。序盤はより少なく、後半は緩やかに。

## 3. ライフ / ☆閾値の設計公式

### 3.1 種別依存公式

| 種別          | ☆3  | ☆2    | ☆1    | life  | ☆1→lifeの余裕 |
| ------------- | --- | ----- | ----- | ----- | ------------- |
| **Teach**     | opt | opt+1 | opt+3 | opt+6 | 3             |
| **Practice**  | opt | opt+2 | opt+4 | opt+6 | 2             |
| **Challenge** | opt | opt+3 | opt+6 | opt+8 | 2             |

`opt` = 最適解（☆3解法）のダメージ値。

### 3.2 設計手順

1. ステージのゴールテキストを設計
2. ☆3解法（最適コマンド列）を決定 → optを計算
3. 種別に応じた公式で☆2/☆1/lifeを算出
4. ☆2解法・☆1解法が実在するか検証（想定解法パスの確認）

### 3.3 具体例

**N18-T（dw導入 Teach）**: opt=2（`w`+`dw`）

```
☆3=2, ☆2=3, ☆1=5, life=8
```

**N32-C（c+TextObj Challenge）**: opt=8

```
☆3=8, ☆2=11, ☆1=14, life=16
```

## 4. 難易度カーブ

### 4.1 段階別opt値

| 段階 | ノード範囲 | opt値 | テキスト行数 |
| ---- | ---------- | ----- | ------------ |
| 序盤 | N01-N08    | 1-3   | 3-8行        |
| 中盤 | N18-N30    | 3-8   | 5-15行       |
| 後半 | N31-N41    | 5-15  | 8-20行       |

### 4.2 カーブ形状

**緩やかな右肩上がり + 波状**

各ノードのTeachステージでopt値が一時的に下がり（新コマンド1回で解ける簡単なステージ）、Practice/Challengeで上がる。全体としては右肩上がり。

### 4.3 分岐間バランス

同階層の分岐ノード（例: N31 [d+TextObj] と N32 [c+TextObj]）はopt値を**±1以内**に揃える。

### 4.4 {n}Gステージ設計制約

`{n}G`（N12）は実務的なコマンドだが、ゲーム内で行番号が見えるため万能移動になりうる。対策:

- N12のステージでは`{n}G`自体の使い方を教える（エラー行ジャンプ等のシナリオ）
- **他ノードのステージでは`{n}G`が最適解にならない設計**にする（変更箇所が近距離、同一行内等）
- `{n}G`は手札に含まれるがopt解法には使わない前提で☆閾値を設計

## 5. 渇望→報酬サイクル

### 5.1 全サイクル一覧（7件）

| #   | 渇望（苦痛）                | 報酬（快感）                   | 配置          | 強度 | 種別     |
| --- | --------------------------- | ------------------------------ | ------------- | ---- | -------- |
| 1   | `x`連打で5文字消す          | `dw`一発                       | N02-P → N18-T | ★★★  | ノード間 |
| 2   | `dl`連打→`i`→打ち直し       | `r`一発で置換                  | N06-T内       | ★    | ノード内 |
| 3   | `0`→`w`連打                 | `^`一発でインデント先頭        | N11-T内       | ★    | ノード内 |
| 4   | `dw`+`i`+入力+`Esc`の4手    | `cw`で2手                      | N27-T内       | ★★   | ノード内 |
| 5   | `ciw`→`w`→`ciw`手動繰り返し | `.`一発リピート                | N32-C → N34-T | ★★★  | ノード間 |
| 6   | `v`→移動連打→`d` (4手)      | `diw`で1手                     | N31-T内       | ★★   | ノード内 |
| 7   | `f`→`df`手動繰り返し        | `;`で次のマッチ、`.`で繰り返し | N24-T / N34-P | ★★   | ノード内 |

### 5.2 実現方法

- **ノード間サイクル**（#1, #5）: 渇望ステージと報酬ステージが別ノード。自然に成立。
- **ノード内サイクル**（#2-#4, #6, #7）: フレーバーテキストで直前の苦痛体験を想起させる。

## 6. テキスト題材ガイドライン

### 6.1 language配分

| language       | 配分 | 用途                                                 |
| -------------- | ---- | ---------------------------------------------------- |
| `plaintext`    | 30%  | 序盤チュートリアル（英単語リスト、TODO、簡単な英文） |
| `css` / `html` | 25%  | 中盤（属性変更、タグ編集等）                         |
| `json`         | 15%  | 中盤（設定ファイル編集）                             |
| `javascript`   | 20%  | 後半（関数修正、変数リネーム等）                     |
| `python`       | 10%  | 後半（インデント操作、関数修正等）                   |

### 6.2 テキスト基準

| 項目               | 基準                                                      |
| ------------------ | --------------------------------------------------------- |
| 言語               | 英数テキスト限定（日本語IME回避）                         |
| 1行の文字数        | 40-80文字（エディタ領域に収まる範囲）                     |
| チュートリアル英文 | プログラミング関連（変数名リスト、TODO、コメント等）      |
| コードのリアリティ | 簡略化した実在風コード。動作不要だが構文は正しく          |
| フレーバーテキスト | ミッション形式（「バグを修正しろ」「CSSの色を変えろ」等） |
