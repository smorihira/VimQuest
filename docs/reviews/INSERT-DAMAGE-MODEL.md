# INSERT モード ダメージモデル設計

> **Status: IMPLEMENTED** — K=5, 矢印無効 (選択肢A), entry=0 統一, paidEntry 廃止

## 1. 現行モデル (commit 9df7c7b → 旧)

### ダメージの流れ

```
[INSERT進入] ──→ [INSERT内操作] ──→ [Esc退出]
  entryDamage        (free)        insertDamage
```

| 経路                                    | entry                          | Esc (nothing typed) | Esc (typed n chars) |
| --------------------------------------- | ------------------------------ | ------------------- | ------------------- |
| **Path A** (i/a/I/A/o/O)                | 0                              | 1                   | ceil(n/5)           |
| **Path B** (s/S/C/cc/ci"/ciw/ct;/cw...) | 1 (parseResult.damage)         | 0 (paidEntry=true)  | ceil(n/5)           |
| **Visual c**                            | 0                              | 1                   | ceil(n/5)           |
| **`.` (dot repeat)**                    | 1 (固定、INSERT session不使用) | —                   | —                   |

### INSERT内の操作コスト

| 操作      | charCount への影響  | ダメージ        |
| --------- | ------------------- | --------------- |
| 文字入力  | +1                  | 0 (Esc時に計上) |
| Enter     | +1                  | 0               |
| Backspace | max(0, charCount-1) | 0               |
| 矢印キー  | 変化なし            | **0**           |

## 2. 問題点

### 問題 A: Path A / Path B の非対称性

同じ「INSERT 1回セッション」なのに、コスト構造が異なる。

```
s → Esc (nothing typed)  = 1 + 0 = 1 ← OK、xと同等
i → Esc                   = 0 + 1 = 1 ← OK
ci" → "blue" → Esc       = 1 + ceil(4/5) = 2
```

**ユーザー指摘**: 全部 entry=0、Esc=min(1) で統一すべきでは？

```
ci" → "blue" → Esc = 0 + ceil(4/5) = 1  (entryも含めて1回のINSERT操作)
s → "a" → Esc      = 0 + ceil(1/5) = 1
```

### 問題 B: 矢印キー移動エクスプロイト（致命的）

INSERT 内で矢印キー移動が無料なので、たった 1 ダメージでファイル内のどこにでも移動できる。

```
i → 矢印×30 → Esc = テキスト変更ゼロ、damage = 1（最低保証）
```

ノーマルモードでの移動は 1 コマンド = 1 ダメージ。
INSERT 矢印は**移動コストの完全なバイパス**。

テキスト差分ベースの計算にしても、差分がゼロなのでこの問題は解決しない。
**問題の本質はテキスト変更ではなくカーソル移動**。

### 問題 C: Backspace による無料削除

Backspace が charCount を 0 以下にできない（`max(0, charCount-1)`）ため、
INSERT 開始前のテキストを Backspace で削除してもコスト 0。

```
i → 矢印で移動 → BS×50（既存テキスト全削除、charCount=0!）→ 新テキスト×50 → Esc
charCount = 50, damage = ceil(50/5) = 10 + 1 = 11
vs ノーマルモードで同じ操作: 50 + 10 = 60 ダメージ以上
```

### 問題 D: リライト（全消し再入力）エクスプロイト

`ceil(n/K)` の除数 K が大きいと、行やファイルを消して打ち直す方が
ノーマルモードで移動+編集するより安くなるケースがある。

```
1234x6789 → 123456789 に修正したい場合:

ターゲット: l l l l s 5 Esc = 4(移動) + 1(INSERT) = 5 damage
リライト:   dd i 123456789 Esc = 1(dd) + ceil(9/5)=2 = 3 damage  ← 勝つ！
```

移動コスト = 1/文字 に対して入力コスト = 1/K文字。
**K が大きいほどリライトが有利になる**。

| K   | dd+9文字リライト | l×4+s+5 | 勝者       |
| --- | ---------------- | ------- | ---------- |
| 5   | 1 + 2 = 3        | 5       | リライト   |
| 3   | 1 + 3 = 4        | 5       | リライト   |
| 2   | 1 + 5 = 6        | 5       | ターゲット |
| 1   | 1 + 9 = 10       | 5       | ターゲット |

K≤2 なら防げるが、正当な INSERT 使用（`ci"` + 短い単語入力）まで高コストになる。
**根本的に `ceil(n/K)` モデルでは解決できない**。

## 3. 決定: フリーワード＋超過課金モデル

### 3.1 設計思想

INSERT モードは**単語レベルの修正**を想定している。
`const`、`blue`、`hello` のような短い単語の入力には大きなダメージを与えたくない。

一方、行全体や文全体を打ち直す「リライトハック」は阻止したい。

→ **最初の K 文字はフリー（セッション基本コスト 1 に含む）、超過分は 1文字 = 1ダメージ**

### 3.2 計算式

```
insertDamage = 1 + max(0, n - K)
```

- `n` = INSERT セッション中に入力した文字数（charCount）
- `K` = フリーワード閾値（定数）
- 最低 1 ダメージ（i→Esc 防止）
- K 文字以内なら 1 ダメージで済む（単語レベルの編集を保護）
- K 文字超過分は 1 文字 = 1 ダメージ（リライトを抑止）

### 3.3 入場コスト統一

> **全経路 entry = 0。Path A / Path B / Visual c すべて統一。**

paidEntry フラグは廃止。

### 3.4 K の選択

INSERT は「1単語を入力する」用途を想定。K は「一般的な単語の長さ」に基づく。

| 言語                 | 平均単語長 | 参考                  |
| -------------------- | ---------- | --------------------- |
| 英語                 | 4.7 文字   | Oxford English Corpus |
| プログラミング識別子 | 6-8 文字   | 変数名・関数名        |
| vim の置換対象       | 3-6 文字   | 短い単語・値          |

**K の候補**:

| K     | 意味              | ci"+"blue"(4) | ci"+"hello"(5) | ci"+"orange"(6) | S+20文字    | dd+9文字    |
| ----- | ----------------- | ------------- | -------------- | --------------- | ----------- | ----------- |
| 4     | 4文字まで無料     | 1             | 1+1=2          | 1+2=3           | 1+16=17     | 1+1+5=7     |
| **5** | **5文字まで無料** | **1**         | **1**          | **1+1=2**       | **1+15=16** | **1+1+4=6** |
| 6     | 6文字まで無料     | 1             | 1              | 1               | 1+14=15     | 1+1+3=5     |
| 7     | 7文字まで無料     | 1             | 1              | 1               | 1+13=14     | 1+1+2=4     |

### 3.5 リライト vs ターゲットの比較（K=5）

**例 1: `1234x6789` → `123456789`（1文字修正）**

| 方法             | 操作               | ダメージ                           |
| ---------------- | ------------------ | ---------------------------------- |
| ターゲット (l×4) | l l l l s 5 Esc    | 4 + 1 = **5**                      |
| ターゲット (fx)  | fx s 5 Esc         | 1 + 1 = **2**                      |
| リライト         | dd i 123456789 Esc | 1 + (1+max(0,9-5)) = 1 + 5 = **6** |

→ **fx がベスト(2)。リライト(6)はターゲット(5)より高コスト。** ✓

**例 2: `let color = "red"` → `let color = "blue"`（ci" で単語置換）**

| 方法           | 操作                     | ダメージ             |
| -------------- | ------------------------ | -------------------- |
| ターゲット     | w w w ci" blue Esc       | 3 + 1 = **4**        |
| ターゲット (f) | f" ci" blue Esc          | 1 + 1 = **2**        |
| リライト (S)   | S let color = "blue" Esc | 1+max(0,22-5)=**18** |

→ **ターゲットが圧勝。S リライトは激高。** ✓

**例 3: 20行ファイル、行 10 の 1 文字変更**

| 方法                | 操作                    | ダメージ                   |
| ------------------- | ----------------------- | -------------------------- |
| ターゲット          | 9j f{char} s X Esc      | 9 + 1 + 1 = **11**         |
| リライト (現在行 S) | 9j S {20文字再入力} Esc | 9 + 1+15 = **25**          |
| 全消し dG           | dG i {全テキスト} Esc   | 1 + 1+max(0,400-5)=**396** |

→ **ターゲットが圧勝。全消しは論外。** ✓

**例 4: `const`→`let` 置換（3文字、K以内）**

| 方法      | 操作        | ダメージ           |
| --------- | ----------- | ------------------ |
| ciw + let | ciw let Esc | 1+max(0,3-5)=**1** |

→ **短い単語の置換は 1 ダメージ。ゲーム体験を損なわない。** ✓

### 3.6 INSERT 内の矢印キー

#### 選択肢 A: 矢印キー無効化

INSERT 内での矢印キーを完全に無効化。

**メリット**:

- エクスプロイト完全防止
- 実装がシンプル（矢印ハンドラを消すだけ）
- vim 的に「正しい習慣」を強制

**デメリット**:

- vim の実際の挙動と異なる（INSERT 中に矢印は使える）
- 初心者が戸惑う可能性

#### 選択肢 B: 矢印キー 1ダメージ

INSERT 内の矢印キー押下ごとに 1 ダメージ加算。

```
insertDamage = 1 + max(0, n - K) + arrowCount
```

**メリット**:

- vim の挙動を維持しつつエクスプロイト防止
- 矢印を使っても動くが、コスト意識が生まれる

**デメリット**:

- 矢印で 1 文字戻って修正するだけで +1 ダメージ（厳しい）
- INSERT 内のダメージがリアルタイムに増加する UI 表現が必要

#### 選択肢 C: 矢印キー有効・無料（現状維持）

矢印キーはそのまま無料で使える。

**メリット**: 初心者に優しい、vim 準拠
**デメリット**: 移動エクスプロイトが残る（1ダメージで無限移動）

#### 最終決定: 選択肢 A（矢印無効化）を採用

**選択肢 A（矢印無効化）** を採用。理由:

1. **エクスプロイト完全防止**: 矢印による無料移動を根本から断つ
2. **実装がシンプル**: 矢印ハンドラを削除するだけ、ダメージ計算への影響なし
3. **Backspace 制限が自然に解決**: 矢印でカーソルを動かせないため、BS で既存テキストを削除する穴も塞がる
4. **vim 教育的に正しい**: VimQuest は vim 操作を学ぶゲーム。INSERT 中に矢印で移動する癖をつけさせない
5. **選択肢 B は UI 複雑化を招く**: INSERT 中にダメージがリアルタイム変動する表現が必要になり、UX が悪化する

vim 実機では INSERT 中に矢印は使えるが、VimQuest はゲームであり、
「ノーマルモードで移動、INSERT は入力のみ」という vim の設計思想を強制する方が教育効果が高い。

### 3.7 Backspace の扱い

#### 最終決定: セッション文字のみ削除可能

矢印無効化（選択肢 A）に伴い、Backspace も自然に制限される:

- `charCount > 0` の場合のみ Backspace を実行（セッション中に入力した文字の削除）
- `charCount === 0` で Backspace を押しても**何も起きない**（既存テキストの削除を防止）
- vim のデフォルト動作（`set backspace=` 未設定時）と一致

これにより問題 C（Backspace による無料削除）が完全に解決される。

## 4. `.` (dot repeat) との整合

現在 `.` は play engine で固定 1 ダメージ。executor 内で INSERT + text + Esc を完結。

フリーワードモデルでも `.` は **1 ダメージ固定で維持**。
理由: `.` は vim の効率化コマンドであり、コスト的に有利であるべき。

## 5. 全体ダメージ表（K=5 採用時）

### INSERT セッション

```
insertDamage = 1 + max(0, n - 5)
```

| 操作                    | n (入力文字数) | damage   | 備考                        |
| ----------------------- | -------------- | -------- | --------------------------- |
| i + Esc                 | 0              | 1        | 最低1 (i→Esc exploit 防止)  |
| s + Esc                 | 0              | 1        | s削除はINSERT前。入力なし→1 |
| i + "abc" + Esc         | 3              | 1        | K以内 → フリー              |
| ci" + "blue" + Esc      | 4              | 1        | K以内 → フリー              |
| ci" + "hello" + Esc     | 5              | 1        | ちょうどK → フリー          |
| ciw + "orange" + Esc    | 6              | 2        | K超過1文字                  |
| S + "hello world" + Esc | 11             | 7        | K超過6文字                  |
| S + 20文字再入力 + Esc  | 20             | 16       | リライトは高コスト          |
| o + "new line" + Esc    | 8              | 4        |                             |
| . (dot repeat)          | —              | 1 (固定) | play engine で直接設定      |

### ノーマルモード（変更なし）

| 操作                  | damage            |
| --------------------- | ----------------- |
| 移動系 (w/j/$/^/f 等) | 1                 |
| 削除系 (x/dd/dw 等)   | 1                 |
| Esc (normal mode)     | 0                 |
| u / Ctrl+R            | rollback / replay |
| ; , n N               | 1                 |

```typescript
interface InsertSession {
```

## 7. 最終決定まとめ (commit f8957b2)

### 採用モデル

```
insertDamage = 1 + max(0, charCount - 5)
```

| 項目                     | 決定                          | 理由                                                                                                                                                                 |
| ------------------------ | ----------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **K (フリーワード閾値)** | **5**                         | 英語の平均単語長(4.7文字)をカバー。`const`(5), `blue`(4), `hello`(5) が全て1ダメージに収まる。K=4 だと `const` が 2 ダメージで不自然。K=6 だとリライト抑止力が弱まる |
| **entry コスト**         | **全経路 0**                  | Path A/B/Visual c で統一。INSERT は「1回のセッション」として Esc 時に一括精算。paidEntry フラグは廃止                                                                |
| **矢印キー**             | **無効化 (選択肢 A)**         | エクスプロイト完全防止 + BS 制限も自然に解決 + vim 教育的に正しい                                                                                                    |
| **Backspace**            | **セッション文字のみ**        | `charCount > 0` でのみ実行。既存テキストの無料削除を防止                                                                                                             |
| **dot repeat**           | **1 ダメージ固定 (変更なし)** | vim の効率化コマンドとしてコスト的に有利であるべき                                                                                                                   |

### 旧モデルからの変更点

| 変更                  | 旧                                          | 新                         | 影響                                                                |
| --------------------- | ------------------------------------------- | -------------------------- | ------------------------------------------------------------------- |
| Esc ダメージ計算      | `ceil(n/5)` (Path A) / `ceil(n/5)` (Path B) | `1 + max(0, n-5)` (全統一) | n≤5: 同じ(1)。n≤6: 同じ(2)。n>6: 新モデルの方が高い（リライト抑止） |
| entry コスト (Path B) | `parseResult.damage` (=1)                   | 0                          | Path B INSERT セッションは全て −1 ダメージ                          |
| paidEntry フラグ      | あり (Path B=true, 他=false)                | **廃止**                   | InsertSession がシンプルに                                          |
| 矢印キー              | 無料で移動可能                              | **無効**                   | 移動エクスプロイト防止                                              |
| Backspace             | charCount=0 以下でも実行可能                | charCount>0 のみ           | 既存テキスト削除防止                                                |

### 解決した問題

- **問題 A** (非対称性): entry=0 統一で解消。全経路が同じ計算式
- **問題 B** (矢印エクスプロイト): 矢印無効化で完全解消
- **問題 C** (BS 無料削除): charCount>0 制限で解消（矢印無効化の副産物）
- **問題 D** (リライトハック): `1+max(0,n-5)` の線形コストで、長い入力ほど高コスト化。`dd+リライト` はターゲット編集より常に高コスト
- **redo バグ** (副次的修正): 旧モデルでは Path B の entry damage が redo 時に復元されなかった。entry=0 により redo のダメージ計算が正確に
