## S1: ツリートポロジー確定

**現状（VimQuest.md セクション2.4）**:  
大まかなツリー構造は示されているが、全ノード・全接続は未確定。

```
現状の記載:
[h j k l] → [x] → [w b e] → [dw / de / db]
                    └→ [0 $] → [d$ / d0]
                               └→ [^]
```

**議論すべき点**:

1. ツリーの全ノード定義（1ノード＝1〜複数コマンドのセット）
2. ノード間の依存関係（前提→後続の矢印）
3. 分岐ポイントと合流ポイント
4. ツリーの深さ（最長パスのノード数）
5. ツリーの幅（同時にavailableになるノード数の最大値）
6. 総ノード数

**制約（確定済み）**:

- v1.0 コマンド数: 約74個（Op8 + Motion30 + TextObj16 + Instant20）
- 移動と編集を交互に導入する原則
- テキストオブジェクトが中盤のハブとなる
- ドットリピート（`.`）が感動の最高峰

**議論**:

### S1-a: 合流ノードの前提条件ルール

- **決定: ALL（全前提クリア必須）** — 学習ゲームのため前提知識の保証が重要。

### S1-b: [i / a] ノードの接続先

- **決定: [x]から分岐** — 削除→挿入の自然な流れ。c系コマンドの前にInsert概念を体験。

### S1-c: [u / Ctrl+R] の扱い

- **決定: 最初から常時利用可能（ノード不要）** — ENGINE-SPECで「ダメージ0」確定済み。安全網は常時提供。

### S1-d: [gg / G / {n}G] の接続先

- **決定: [0 $]から分岐** — 行内移動→ファイル全体移動への範囲拡大。

### S1-e: テキストオブジェクトの粒度

- **決定: 2段階** — [iw/aw]（単語）→ [残り14個]（デリミタペア）。概念導入→パターン展開。

### S1-f: {count} の前提ノード

- **決定: [f/t]の後** — テキストオブジェクト前にj連打→5jの渇望サイクルを体験。

### S1-g: [gj / gk] の前提ノード

- **決定: [gg/G]の後** — ファイル移動系の拡張。

### 追加帰結（S1-a〜gから派生）

- [df/dt/cf/ct]を[df/dt]と[cf/ct]に**分割**（cはInsertモードに入るため[i/a]前提が必要）
- [d+textobj]は[delimiter TextObj] + [dw/de/db]の**合流ノード**
- [c+textobj]は[delimiter TextObj] + [cf/ct]の**合流ノード**
- 合流ノード（ALL）: 全5箇所

**決定**: 全41ノード。詳細は以下のノード一覧表とエッジリストを参照。

## S4: ライフ / ☆閾値の設計公式

**現状（VimQuest.md セクション2.1）**:

- ☆3解法 → ☆2解法 → ☆1解法をまず設計
- ☆3閾値 = ☆3解法ダメージ
- ☆2閾値 = ☆2解法ダメージ
- ☆1閾値 = ☆1解法ダメージ
- ライフ = ☆1解法 + 余裕2-3

**議論**:

### 種別依存の公式（確定）

| 種別          | ☆3  | ☆2    | ☆1    | life  | ☆1→lifeの余裕 |
| ------------- | --- | ----- | ----- | ----- | ------------- |
| **Teach**     | opt | opt+1 | opt+3 | opt+6 | 3             |
| **Practice**  | opt | opt+2 | opt+4 | opt+6 | 2             |
| **Challenge** | opt | opt+3 | opt+6 | opt+8 | 2             |

- `opt` = 最適解（☆3解法）のダメージ値
- Teachは余裕を大きくして初心者が詰まらない設計
- Challengeは☆間の差を広げて複数解法の幅を表現

**決定**: 上記の種別依存公式で確定。
