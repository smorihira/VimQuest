# Undo / ダメージ回復 設計議論

## 1. 現状の挙動

### 実装（既にVim準拠の部分）
- **移動コマンド**（h,j,k,l,w,b,e…）は undoStack に**積まれない** ✅
- `u` は**テキスト変更のみ**を巻き戻す（Vim準拠） ✅
- テキスト・カーソル・モードが正しく変更前に復元される ✅

### 問題: ダメージ回復量が足りない
- `u` 実行時、`op.damage`（= 変更操作の damage:1）しか回復しない
- 変更後に移動した分のダメージが取り残される

### シナリオ: `x → j → l → u`

| 手 | 操作 | damage | undoStack | テキスト変更 |
|----|------|--------|-----------|------------|
| 1 | `x`（削除）| 1 | [x-op(damage:1)] | ✅ 文字削除 |
| 2 | `j`（移動）| 2 | （変化なし） | なし |
| 3 | `l`（移動）| 3 | （変化なし） | なし |
| 4 | `u`（undo）| **2**（3 - 1 = 2） | [] | ✅ 文字復活 |

**問題**: テキストとカーソルは `x` の前に正しく戻る。しかしダメージは 3→2 で、
`j`, `l` の移動ダメージ（2）が回復されない。カーソルは変更前の位置に戻っているのに、
移動で食らったダメージだけが残る → 不自然。

### 期待される挙動
- `u` でテキストが `x` の前に戻ると同時に、**damage も `x` を打つ前の状態（0）に巻き戻る**べき
- 「あの変更をする前の状態に完全に戻る」のが自然な体験

---

## 2. 移動のundoについて（検討結果: 実装しない）

### 検討: 移動だけの `u` （例: `j → u` で `j` を巻き戻す）を実装すべきか？

**結論: 実装しない（Vim準拠）**

理由:
1. **Vim準拠が最優先** — VimQuestの目的はVimを学ぶこと。`u` に独自挙動を持たせると学習効果が落ちる
2. **逆操作が常にある** — `j→k`, `w→b`, `l→h`。移動ミスは逆方向で自然に修正可能
3. **`:r` リセットがある** — 致命的なミスはリセットで対応可能
4. **undoスタック汚染の回避** — 移動をundoableにすると、テキスト変更に辿り着くまでの `u` 連打が必要になり、本末転倒

---

## 3. 設計方針

### 方針: `damageAtEntry` による完全巻き戻し

**コンセプト**: 「undoは最後のテキスト変更の直前の状態にダメージも完全巻き戻す」

#### 挙動

1. undoスタックにはテキスト変更のみ積む（現状のまま）
2. 各Operation に `damageAtEntry` を記録（その変更を行う直前の累計damage）
3. `u` 実行時:
   - テキスト＋カーソルを変更前に復元（現状のまま）
   - damageを `damageAtEntry` に巻き戻す（変更+途中移動も全回復）← **ここが変更点**
4. `Ctrl+R` 実行時:
   - damageをredoされた操作のdamage分だけ加算（=変更のdamage 1のみ）
   - 移動分は戻らない（移動はredoで再実行されない）

#### シナリオ: `x → j → l → u`

| 手 | 操作 | damage | undoStack |
|----|------|--------|-----------|
| 1 | `x`（削除）| 1 | [{..., damageAtEntry: 0}] |
| 2 | `j`（移動）| 2 | （変化なし） |
| 3 | `l`（移動）| 3 | （変化なし） |
| 4 | `u`（undo）| **0**（damageAtEntry=0 に巻き戻し） | [] |

✅ テキスト変更も移動ダメージも全て巻き戻され、「やり直し」の体験として自然

#### シナリオ: `j → j → x → j → u`

| 手 | 操作 | damage | undoStack |
|----|------|--------|-----------|
| 1 | `j` | 1 | |
| 2 | `j` | 2 | |
| 3 | `x` | 3 | [{..., damageAtEntry: 2}] |
| 4 | `j` | 4 | |
| 5 | `u` | **2**（damageAtEntry=2 に巻き戻し） | [] |

✅ `x` の前にした移動(damage 2)は残る。`x` 後の移動(damage 1)は回復される

#### シナリオ: `dw → j → ci" → hello → Esc → u`

| 手 | 操作 | damage | undoStack |
|----|------|--------|-----------|
| 1 | `dw` | 1 | [{..., damageAtEntry: 0}] |
| 2 | `j` | 2 | |
| 3 | `ci"hello Esc` | 3 | [{..., damageAtEntry: 0}, {..., damageAtEntry: 2}] |
| 4 | `u` | **2**（ci"の damageAtEntry=2 に巻き戻し） | [{..., damageAtEntry: 0}] |
| 5 | `u` | **0**（dw の damageAtEntry=0 に巻き戻し） | [] |

✅ 一つずつ変更を巻き戻し、各変更の「前」の状態に完全復帰

## 4. 懸念と対策

### 懸念1: 「探索してからundoで全回復」が強すぎないか？

> `dw → jjjjjjjj → u` で移動8ダメージも全回復される

**対策案:**
- (a) 許容する — undoは安全網。探索コストを気にせず試行錯誤させるのがゲームの哲学
- (b) undo時に移動ダメージの半分だけ回復（ペナルティ）
- (c) undo回復に上限を設ける

**推奨: (a) 許容する**
- 仕様書に「死を恐れない試行錯誤を保証する」とある
- 探索にはリアル時間がかかる → 高スコアを狙うプレイヤーは自然と効率的に動く
- undoで戻しても「やり直す」のにまたdamageがかかる

### 懸念2: 移動ミスだけ戻したい時はどうする？

> `j` で間違った方向に移動。テキスト変更は戻さず移動だけ戻したい

**現実**: 実際のVimでもこれは `k`（逆方向）で解決する。undoではない。
ゲームでも `k` を押せば良い（damage +1 だが）。

**許容理由**: 移動は必ず逆操作がある（h↔l, j↔k, w↔b, gg↔G）。ゲーム性として「逆移動のダメージ」を払わせるのは妥当。

## 5. SPELL記録への影響

### 現状のSPELL仕様（変更不要）
> SPELL = クリア時点のundoスタック上の操作列

案Xでは undoStack にテキスト変更のみが積まれるため:
- SPELLには変更操作のみが記録される ✅
- 移動はSPELLに含まれない（もともとスタックに入らないため）
- Result画面の YOUR SPELL には意味のある操作だけが並ぶ ✅

### ダメージ表示
- DMG カウンターは全操作（移動含む）の累計 → 変更なし
- SPELL の各エントリの damage は Operation.damage (=1) → 変更なし
- SPELLの合計とDMGは一致しなくなる（移動分がある）が問題なし

---

## 6. 決定事項

- [x] 移動は undoStack に積まない（現状のまま。Vim準拠）
- [x] 移動のundo機能は実装しない（`:r` リセットで対応）
- [ ] `u` 実行時、`damageAtEntry` まで完全巻き戻し（変更+途中移動のダメージ全回復）

---

## 7. 実装変更点

### types/editor.ts
- `Operation` 型に `damageAtEntry: number` を追加

### commandExecutor.ts
- `pushUndo` に `damageAtEntry` パラメータを追加
- 各呼び出し元から累計damageを渡す
- 移動の扱いは変更なし（もともとundoStackに積んでいない）

### usePlayEngine.ts
- undo処理: `setDamage(op.damageAtEntry)` に変更（現状の `prev - op.damage` から）
- redo処理: `setDamage(prev + op.damage)` は変更なし
- `executeCommand` 呼び出し時に現在の `damage` を渡すインターフェース追加

### SPELL記録
- 変更なし（もともとundoStackから構築、移動は入っていない）
