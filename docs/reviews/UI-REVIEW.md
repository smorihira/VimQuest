## 🔴 Priority 1: 画面横断の構造問題

### 1.1 ナビゲーション・画面遷移フローの統一
- **現状の問題**: 各画面のヘッダー/戻る導線がバラバラ
  - スキルツリー: ヘッダーあり（進捗表示）
  - プレイ画面: ヘッダーなし
  - リザルト: ボタンのみ（もう一度 / 次のステージへ）
  - ゲームオーバー: ボタンのみ（ツリーに戻る / リトライ）
  - 武器習得: ボタンなし（キー入力で進む）
  - ランディング: ヘッダーなし
- **議論ポイント**:
  - 全画面に共通ヘッダーを入れるか？
  - 画面遷移フロー: Landing → Tree → Play → Result/GameOver → (Weapon) → Tree の流れをどう表現するか
  - プレイ中に「ツリーに戻る」「ギブアップ」導線は必要か
- **決定**: ✅ B案ベース採用
  - プレイ画面: ヘッダーなし（没入感優先）
  - Result/GameOver/Weapon: オーバーレイ的扱い、ヘッダーなし
  - Skill Tree: ヘッダーあり（ロゴ+進捗表示、ホーム画面的位置付け）
  - Landing: ヘッダーなし（1画面完結）
  - プレイ中のギブアップ: 左上に控えめアイコン + `:q!` 入力でツリーに戻れるイースターエッグ
  - Result: 「もう一度」「次のステージへ」「ツリーへ」の3択
  - GameOver: 「リトライ」「ツリーに戻る」の2択
  - Weapon: キー入力で自動遷移

### 1.2 画面遷移フロー図
- **現状の問題**: 仕様書に画面遷移の全体図がない
- **議論ポイント**: 遷移パターンの洗い出し
  - Landing → Tree（はじめる/つづける）
  - Tree → Play（ステージ選択）
  - Play → Result（クリア時）
  - Play → GameOver（ライフ0）
  - Result → Weapon（新武器アンロック時のみ）
  - Result → Tree / Result → Play（次 or リトライ）
  - GameOver → Play（リトライ）/ GameOver → Tree
  - Weapon → Tree / Weapon → Play（次ステージ）
- **決定**: ✅ 以下の遷移フローで確定
  - Landing → Tree（はじめる/つづける）
  - Tree → Play（ステージ選択）
  - Play → Result（クリア時）
  - Play → GameOver（ライフ0）
  - Result → Play（次ステージ直接遷移OK）/ Result → Tree
  - Result → Weapon（新武器アンロック時のみ）→ Tree（ツリーで開放状態を見せる）
  - GameOver → Play（リトライ）/ GameOver → Tree
  - Landingの再訪問導線は不要。Treeがホーム。Landingは初回 or URL直アクセス時のみ

## 🟡 Priority 2: 画面別の設計問題

### 2.1 Result画面: コマンド数の齟齬
- **現状の問題**: COMMANDS: 5 だが YOUR SPELL に4要素しか表示されていない
- **議論ポイント**:
  - Esc はコマンドカウントに含めるか？（仕様上 Esc はダメージ0）
  - テキスト入力 `#333333` はスペルとして1つにカウントするか、文字数ベースか
  - そもそも COMMANDS 数 = ダメージ数 ではない（Insert入力分の差）ため表示ルールの明確化
- **決定**: ✅ ラベルを `COMMANDS` → `DAMAGE` に変更。YOUR SPELLの各要素下に小さくダメージ数（-1, -0等）を表示。LIFE（残り/最大）も併記。

### 2.2 Result画面: 効率コマンドの強調ルール
- **現状の問題**: `ci"` だけ緑背景で強調されているが、ルールが不明
- **議論ポイント**:
  - 「そのステージで求められるキーコマンド」を強調するのか
  - 全コマンド同色でシンプルにするのか
  - 色分けのルール（新武器=強調、移動=薄め、など）
- **決定**: ✅ ステージの学習対象コマンドを緑（#44ddaa）で強調。それ以外は通常accent色。Esc/テキスト入力はdim色。

### 2.3 Result画面: 星表示の混乱
- **現状の問題**: 今回の星が ★★☆、BESTが ★★★ と小さく表示。視認性・意味の区別が弱い
- **議論ポイント**:
  - BEST表示は必要か？（モチベーション的にリトライ意欲を掻き立てる？）
  - 今回とBESTの見せ方をどう差別化するか
- **決定**: ✅ 大きな星は今回の結果のみ。BESTは星ではなく `BEST: 3 damage` のテキスト表示。BEST更新時は `NEW BEST!` を金色で追加。

### 2.4 Skill Tree: レイアウトの偏り
- **現状の問題**: ノードが左上に集中、右半分・下半分が空白
- **議論ポイント**:
  - 横スクロール前提でOKか？
  - 自動フィット（ズーム調整）するか
  - ツリーが大きくなったとき（v1.0の60コマンド全配置時）のレイアウト戦略
  - ミニマップの必要性
- **決定**: ✅ 初期表示は全ノード自動フィット。マウスドラッグでパン、ホイールでズーム。ミニマップはv1.0不要。availableノードにフォーカスするボタン [🎯] を用意。レイアウトは左→右、ブランチは下に伸びる構造を維持。

### 2.5 Skill Tree: locked ノードの視認性
- **現状の問題**: ロック状態のノードが薄すぎて読みにくい
- **議論ポイント**:
  - ロックノードのコマンド名は見せるべきか？（「次に何があるか」の期待感 vs 発見のサプライズ）
  - 見せる場合、どの程度の濃さにするか
- **決定**: ✅ コマンド名は見せる。opacity 0.35程度で「読めるが薄い」状態に。渇望感の演出として重要。

### 2.6 Game Over: ログの畳み込み
- **現状の問題**: `j` が3行連続で並ぶ → 冗長
- **議論ポイント**:
  - 繰り返しコマンドは `j ×3 (-3)` のように畳むか
  - 畳むと「浪費の具体性」が薄れるリスク
  - waste操作だけ展開、OK操作は畳む折衷案は？
- **決定**: ✅ 連続する同一コマンドは種類問わず畳む（`j ×3 -3`）。waste操作は赤ハイライト維持。一貫性のある表示。

### 2.7 Landing: Vim未経験者への伝達
- **現状の問題**: `ci"` デモは Vim を知らない人には意味不明の可能性
- **議論ポイント**:
  - ターゲットユーザーはVim初心者（ある程度知っている）か完全未経験者か
  - アニメーション付きデモ（ステップ実行）にするか
  - テキスト説明を添えるか
- **決定**: ✅ ターゲットはVim初心者と割り切り。デモ下に `ci" = "" の中身を丸ごと変更` 程度の短いキャプション1行を追加。ステップアニメーションはv1.0不要。

## � Round 2: モック修正後の追加問題点（2026-05-23）

- **決定**: ✅ ボタンラベルは「次のステージへ→」のまま。NEW WEAPON時は内部的にWeapon画面を挟む。バナーはResultに残し期待感を演出。

### R2.3 Landing: 「はじめる」vs「つづける」

### R2.6 GameOver: 合計ダメージの不在
- **現状の問題**: ログで各操作のコストは見えるが、合計ダメージ / ライフ上限の情報がない。「あと何手節約すればクリアできたか」が分からない
- **議論ポイント**:
  - ログ下にサマリー行（`合計ダメージ: 14 / ライフ: 10 → 4オーバー`）を追加するか
  - HINTの「計3ダメージ」と対比できるようにするか
- **決定**: ✅ ログ下にサマリー行を追加（`合計ダメージ: 14 / ライフ: 10 → 4オーバー`）。HINTの最適解ダメージと対比して改善幅を可視化。

### R2.7 Play: エディタ領域とカード領域のバランス

### R2.8 Play: カードの情報密度

- **決定**: ✅ カードにはコマンド名のみ大きく表示、ラベルはホバーツールチップ。手札は最大12〜15枚想定。2行表示可（上段=動詞・モーション、下段=テキストオブジェクト・合成）。

### R2.9 Play: ステージタイトルの視認性

- **決定**: ✅ ステージタイトルとモード表示の配置を整理。実装時にTop bar内でレイアウト調整。

## 🔵 Round 3: インタラクション設計（2026-05-23）

### R3.1 コマンド入力のフィードバック
- **議論ポイント**: `3j` や `diw` のように途中状態がある入力をどう見せるか
  - カウント待ち（`3` → モーション待ち）
  - オペレータ待ち（`d` → 対象待ち）
  - 複合待ち（`3d` → 対象待ち）
- **案A**: Vimライク — エディタ下部にステータスラインで入力中シーケンス表示
- **案B**: カード連動 — 動詞カード浮上 + 有効対象カードのみアクティブ化 + カウント×Nバッジ
- **案C**: 両方（ステータスライン + カード連動）
- **決定**: ✅ 案C採用。教育面（Vimのステータスライン表示を学ぶ）とゲーム面（カード連動で次に何を押せるか可視化）の両立。`d` 押下時に対象カードだけ光る体験は「動詞+対象の合成」学習コンセプトの可視化そのもの。

### R3.2 カード合成の演出
- **議論ポイント**: `d` + `iw` → `diw` のカード合体をどう演出するか
- **案A**: スライド合体 — `iw` が `d` の位置にスライド → 合体 → フラッシュして消える
- **案B**: 中央合流 — 2枚が画面中央に移動 → 合体 → エフェクトがエディタに飛ぶ
- **案C**: インプレース変化 — `d` 押下でパルス維持 → `iw` 押下で `d` が `diw` に変化＋ `iw` フェードアウト → `diw` がエディタに飛んで効果適用
- **決定**: ✅ 案C採用。「動詞が対象を食べて合成コマンドになる」概念を可視化。動詞の位置に合成結果が残り、合成の主体が動詞であることが直感的に学べる。全体0.3〜0.5秒。合成完了時に打鍵音再生。

### R3.3 カーソル表示
- **議論ポイント**: モード別カーソル形状、Visualモードの背景色、選択範囲の色、カーソル点滅の有無
- **決定**: ✅ Normal=ブロック（█）、Insert=ライン（｜）、Visual=ブロック+選択範囲ハイライト（実Vimと一致）。Visual背景色: `#2a1a3e`（暗い紫）。選択範囲: `rgba(124,124,255,0.25)`（accent半透明）。カーソル点滅なし、代わりに微かなグロー（発光）で位置強調。

### R3.4 Spaceビジョン切替
- **議論ポイント**: ホールド中のゴール表示の切替演出
- **案A**: パッと即座切替。シンプルだが差分が分かりにくい
- **案B**: クロスフェード（0.15秒）。切替感はあるが差分把握は難
- **案C**: 差分ハイライト付き即時切替。変更箇所を0.2秒フラッシュで視線誘導
- **案D**: 透視エフェクト。重なりが読みにくい可能性
- **決定**: ✅ 案C採用。Space押下で即座にゴールテキストに切替 + 変更箇所を0.2秒success色フラッシュ。離すと即座に現在テキストに戻る（フラッシュなし）。「どこを変えればいいか」の気づきを促進。

### R3.5 画面遷移アニメ
- **議論ポイント**: 各画面間のトランジション方針
- **案A**: 統一フェード（0.3秒）。一貫性あるが味気ない
- **案B**: 遷移別カスタム。実装コスト高
- **案C**: フェード（0.2秒）+ ワンポイント
- **決定**: ✅ 案C採用。基本は全遷移共通で暗転フェード（0.2秒）。例外1: Play→Resultのみ暗転中にSTAGE CLEARテキストを0.5秒表示。例外2: Play→GameOverのみ赤フラッシュ後フェード。感情のピークだけ差別化。

### R3.6 リザルト演出の登場順序
- **議論ポイント**: STAGE CLEAR → 星 → SPELL → Stats の登場タイミング
- **案A**: 全部同時。シンプルだが情報過多
- **案B**: シーケンシャル登場（全体2.5秒）+ スキップ可能
- **案C**: 星だけ演出 + 残り同時
- **決定**: ✅ 案B採用。暗転中STAGE CLEAR(0.0s) → ステージ名(0.5s) → 星1つずつ点灯+効果音(0.8s) → YOUR SPELLスライドイン(1.5s) → Statsフェードイン(2.0s) → NEW WEAPONバウンス(2.2s) → ボタン群(2.5s)。クリック/キーで即全表示スキップ可能。

### R3.7 モバイルアクセス時の表示
- **議論ポイント**: デスクトップ専用だがスマホで開いた場合
- **案A**: 何もしない。UX最悪
- **案B**: ブロック画面のみ。明確だが機会損失
- **案C**: 簡易紹介 + ブロック。Landing相当の訴求を見せつつPCへ誘導
- **決定**: ✅ 案C採用。viewport幅768px未満 or タッチデバイスでLanding相当の紹介 + 「⌨️ PCでアクセスしてください」メッセージ。URLコピーボタン付き。ゲーム本体は読み込まない。SNS等からのモバイル流入をPCへ転換。

### R3.8 u（undo）の扱い
- **議論ポイント**: カード化するか、ライフ消費するか、救済措置か
- **案A**: undoなし。緩張感あるが初心者が詰む
- **案B**: undo = 1ダメージ消費。戦略的選択になる
- **案C**: undo無料・無制限。ライフも復元。前の操作に完全に巻き戻る
- **方針決定**: ✅ 案C採用。`u` は正当なVim技術。undoでライフも復元し、1つ前の操作に完全に戻る。何度でもやり直せる。失敗に対するUXを良好に保つ。
- **UI詳細**: ✅ 決定済み
  - テキスト: 1手前に巻き戻り + 変更箇所success色フラッシュ
  - ライフゲージ: 消費分が回復アニメ（ゲージが伸びる）
  - カード: 合成カードは分解して元の2枚に戻る（合成の逆再生、0.2秒）
  - ステータスライン: `u` 表示後、巻き戻されたコマンドを取り消し線で一瞬表示
  - SPELL記録: undoされた操作は消える。最終結果のみ記録（「磨き上げた解法」を称える）
  - 連続undo: 可能。初期状態まで全て戻せる
  - Ctrl+R (redo): undoした操作をやり直せる。実Vimと同じ

### R3.9 不正コマンド入力のフィードバック

## 🟣 Round 4: 仕様の穴埋め（2026-05-23）

### R4.1 Insertモード中のカード領域
- **議論ポイント**: Insertモードではコマンドカードは使えない。30%のカード領域をどうするか
- **案A**: 全カードグレーアウト。画面の30%が無駄
- **案B**: カード領域を畳んでエディタ拡大。画面がガクッと変わる

- **決定**: ✅ 案C採用。70:30固定維持。カード領域にInsertモード情報を表示（入力文字数 `3/5 FREE` → `6/+1 DMG` の進行表示 + Escリマインダー）。「あと何文字無料か」が初心者に重要な情報。

### R4.2 検索（`/`）のUI
- **議論ポイント**: `/` 押下後の検索入力の表示場所、マッチハイライト
- **案A**: ステータスライン流用。実Vimと同じ位置。追加コンポーネント不要
- **案B**: VS Code風オーバーレイ。実Vimと異なる
- **決定**: ✅ 案A採用。`/` 押下でステータスラインが検索入力モードに切替わり、`/keyword` を表示。Enter確定、Escキャンセル。全マッチ箇所を `rgba(255,215,0,0.3)` で背景ハイライト、カーソル上のマッチはより濃く。`n`/`N` で移動。

### R4.3 ステージ開始シーケンス
- **議論ポイント**: ステージに入った瞬間の演出
- **案A**: 即プレイ。最速だが初見で分かりにくい
- **案B**: ステージイントロ（1.5秒）。スキップ可
- **案C**: カウントダウン。速度ゲームっぽくテンポが合わない
- **決定**: ✅ 案B採用。フェードイン後にステージ名+目標をエディタ中央に1.5秒表示。任意キー/クリックでスキップ即開始。「何を目指すか」を先に見せる教育設計。

### R4.4 `.`（ドットリピート）の可視化
- **議論ポイント**: `.` 実行時のカード演出
- **案A**: 専用カードのみ。ホバーで前回コマンド表示
- **案B**: ゴースト再生。前回のカード合成演出を半透明で高速再生（0.15秒）
- **案C**: 演出なし。シンプルだが学習効果薄
- **決定**: ✅ 案B採用。`.` カードは手札にあり、ホバーで「前回: ciw」表示。押下時に前回コマンドのカード合成をゴースト（半透明）で0.15秒高速再生。「何が繰り返されたか」が視覚的に分かる。

### R4.5 レジスタカードの表示
- **議論ポイント**: 仕様書で「レジスタの中身をカードとして表示」とある。具体的なUI。
- **決定**: ✅ カード領域右端に独立グループとして配置。star色（#ffd700）の枠で通常カードと区別。カード上部にレジスタ名（`"a`）、下部に中身プレビュー（最大8文字、超過時省略）。ホバーで全文表示。`"ap`でカードが飛んでペースト、`"ayiw`で中身更新。`"0`レジスタも同様に表示。

### R4.6 シンタックスハイライト
- **議論ポイント**: エディタ内のコードに構文色付けはする？しない？
- **案A**: フルハイライト。言語別にキーワード・文字列・数値・コメント等を色分け
- **案B**: 最小限（コメント・文字列・数値のみ）
- **案C**: ハイライトなし
- **決定**: ✅ 案A採用。正規表現ベースの軽量トークナイザでフルハイライト。v1.0対象言語: CSS, HTML, JavaScript, JSON, Markdown, Python。配色: キーワード=accent紫(#7c7cff)、文字列=success緑(#44ddaa)、数値=star金(#ffd700)、コメント=グレー(#666)。「本物のエディタ」感を出し、コード構造の把握を助ける。

### R4.7 デスクトップ画面サイズ対応
- **議論ポイント**: 最小対応幅・最大幅・レスポンシブ戦略
- **決定**: ✅ 最小対応幅1024px。推奨1280〜1920px。コンテンツ領域max-width:1400pxで中央寄せ（ウルトラワイド対策）。1024px未満はモバイルと同じブロック画面（「画面を広げてください」）。Playのエディタ:カード=70:30は維持。

### R4.8 ローディング / スプラッシュ
- **議論ポイント**: アプリ初回読み込み中の表示
- **決定**: ✅ 背景#1a1a2e。中央にVimQuestテキストロゴ＋ナビゲーターキューブのCSS回転アニメーション（ローディングインジケータ兼ブランドアイコン）。index.htmlのインラインCSS/HTMLで実装（JSロード前に表示）。通常1-2秒。
