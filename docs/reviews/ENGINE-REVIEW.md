# VimQuest エンジン仕様レビュー & 議論

UI仕様の確定後、実装に入る前にゲームエンジン内部の仕様ギャップを洗い出し・議論するためのドキュメント。

---

## 🔴 BLOCKER: 実装前に決めるべき事項（8件）

### B1. テキスト比較アルゴリズム（ステージクリア判定）
- **問題**: ステージクリアの「完全一致」の定義が曖昧
- **未定義項目**:
  - 末尾空白の扱い（`"hello "` vs `"hello"`）
  - 改行コード（LF固定？CRLF許容？）
  - 最終行の末尾改行の有無
  - Unicode正規化（é vs e+accent）
  - カーソル位置は比較対象か（テキストのみ？）
- **影響**: ステージ設計者が信頼できるステージを作れない
- **議論**: 厳密一致 vs 緩和（末尾空白無視等）→ 厳密一致で合意
- **決定**: ✅ `bufferText === goalText` の単純文字列比較。改行=LF固定、末尾空白も比較対象、最終行は改行なし、Unicode正規化なし、比較対象はテキスト内容のみ（カーソル位置はB6の拡張条件）、コマンド実行ごとに判定。ステージデータ側がゴールテキストを正確に定義する責任を持つ。

### B2. オペレータ+モーション有効性ルール
- **問題**: `d`に`w`,`iw`,`$`が有効とあるが、完全なホワイトリストが存在しない
- **未定義項目**:
  - オペレータごとの有効なモーション/テキストオブジェクト一覧
  - オペレータ同士の組み合わせ（`dd`=行削除 は特殊形か）
  - `gu`/`gU`/`g~`等のg系オペレータの対象ルール
  - カード合成UIでどのカードをハイライトするかの判定基準
- **影響**: カード合成ロジック（コアUI）の実装根拠が不足
- **議論**: 3カテゴリ分類制（オペレータ/モーション/テキストオブジェクト）＋即時コマンドで合意
- **決定**: ✅ コマンドを4カテゴリに分類。①オペレータ（d,c,y,gu,gU,g~,>,<）はモーション/テキストオブジェクトを待つ。②モーション（w,b,e,0,$,gg,G,hjkl,f/t,/,%）はカーソル移動＋オペレータ対象。③テキストオブジェクト（iw,aw,i",a"等）はオペレータ対象のみ（Visualモード中は選択範囲として単独使用可）。④即時コマンド（r,s,x等）は単独完結。組み合わせルール: Op+Motion=有効、Op+TextObj=有効、Op×2同キー=行操作(dd,cc等)、Op+Op異キー=無効。数値プレフィックスは全カテゴリに付与可。カードUI: オペレータ押下→有効な対象カードのみハイライト。

### B3. 検索（`/`）の実装仕様
- **問題**: 検索の内部動作が未定義
- **未定義項目**:
  - 正規表現 vs リテラル文字列マッチング
  - 大文字小文字の区別（デフォルト）
  - エスケープシーケンス（`\.`, `\n`等）の対応範囲
  - パターン不正時のエラー処理
  - マッチゼロ時の挙動
  - EOF時のラップアラウンド（`n`で先頭に戻るか）
  - `/`入力中のリアルタイムハイライト有無
- **影響**: 検索エンジンの実装不可
- **議論**: 正規表現 vs リテラル → リテラルで合意（予測可能性優先）
- **決定**: ✅ リテラル文字列マッチ（正規表現なし）。case-sensitive。ラップアラウンドあり（EOF→先頭）。マッチゼロ時はステータスラインに`(pattern not found)`表示、ダメージなし。`/`入力中はステータスラインにパターン表示、リアルタイムハイライトなし（Enter確定後にハイライト）。`n`/`N`=次/前マッチ移動、ダメージ1。オペレータ対象として使用可（`d/pattern`等）。

### B4. undo/redo/SPELLの状態遷移
- **問題**: undo→新コマンド実行時のredoスタック、SPELLの正確な定義が曖昧
- **未定義項目**:
  - undo後に新コマンド実行 → redoスタックはクリアされるか（標準エディタ挙動）
  - SPELLの「クリア時点の操作列」= undoスタック上の操作のみ？
  - undo自体はSPELLに記録されるか
  - redo自体はSPELLに記録されるか
  - 例: `dw → dd → u → u → cw → hello → Esc` のSPELLは？
- **影響**: SPELL表示ロジックとResult画面の正確性
- **議論**: 標準エディタ準拠のundoツリーで合意
- **決定**: ✅ undo後に新コマンド→redoスタッククリア（標準挙動）。undo/redo自体はSPELLに記録しない（メタ操作）。SPELL=クリア時点のundoスタック上の操作列。ダメージもundo/redo=0、undoで戻された操作のダメージは復元。例: `dw→dd→u→u→cw+hello→Esc` → SPELL=[cw→"hello"]、ダメージ=3(dw1+dd1+cw1、u×2は0)。

### B5. Insertモード中のダメージ計算具体例
- **問題**: 文字数閾値制（5文字FREE、6-10で+1、11-15で+2...）の適用例が不足
- **未定義項目**:
  - `ciw` + 3文字入力 + Esc = ダメージいくつ？（ciw=1、入力FREE → 合計1？）
  - `i` + 11文字入力 + Esc = ダメージいくつ？（i=1、11文字→+2 → 合計3？）
  - `s` + 3文字入力 + Esc = ？
  - `O` + 0文字入力 + Esc = ？（空行挿入のみ）
  - バックスペースは文字数を減らすか？
  - Enterは1文字としてカウント？（R5.2で決定済み: Yes）
- **影響**: ステージ難易度設計とライフ計算の正確性
- **議論**: 具体例テーブルで全パターン確認・合意
- **決定**: ✅ Insert系コマンド=1ダメージ+入力文字（5文字FREE、6-10で+1、11-15で+2…）。オペレータ合成（ciw等）も合成全体で1ダメージ。バックスペースは文字数から減算（最終文字数で判定）。Enter=1文字。O+0文字+Esc=1ダメージ。例: `i`+5文字=1、`i`+6文字=2、`ciw`+7文字=2、`s`+3文字=1、`A`+20文字=4。

### B6. 拡張クリア条件のデータフォーマット
- **問題**: カーソル位置・レジスタ条件のステージデータ上の表現方法が未定義
- **未定義項目**:
  - カーソル位置の形式（line/col、0-indexed?）
  - レジスタ条件の指定方法（named/unnamed、完全一致？）
  - ステージJSONスキーマへの組み込み方
  - 複数条件の論理関係（AND?）
  - 拡張条件の表示方法（プレイヤーに事前に見せるか）
- **影響**: 後半ステージの設計不可
- **議論**: JSONスキーマ案で合意
- **決定**: ✅ `clearConditions`オブジェクトで定義。`text`(boolean,デフォルトtrue)=テキスト一致、`cursor`({line,col} 0-indexed)=カーソル位置、`registers`({name:content})=レジスタ内容。すべてAND評価。省略時は該当条件不問。拡張条件があるステージはイントロでアイコン提示（🎯カーソル/📋レジスタ）、プレイ中ステータスラインで残り条件表示。v1.0は大半text のみ、後半数ステージでcursor使用程度。

### B7. LocalStorageエラーハンドリング
- **問題**: ストレージ障害時の挙動が未定義
- **未定義項目**:
  - プライベートブラウジングでLocalStorage無効の場合
  - 容量超過（5-10MB上限）時の処理
  - 保存済みJSONが破損している場合のリカバリ
  - セーブデータのバージョン互換（将来のアップデート時）
- **影響**: アプリクラッシュやデータ喪失のリスク
- **議論**: データ喪失よりアプリ継続優先で合意
- **決定**: ✅ LS無効時→インメモリフォールバック+「進捗は保存されません」バナー。容量超過→古いデータ削除して再試行、失敗→バナー警告。JSON破損→該当キー削除して初期値リセット、全損→全リセット+通知。保存データに`version`フィールドを持たせ、読み込み時にマイグレーション。変換不能→リセット。

### B8. シンタックスハイライトの言語指定
- **問題**: 6言語対応（CSS/HTML/JS/JSON/Markdown/Python）だが、言語判定方法が未定義
- **未定義項目**:
  - ステージデータに`language`フィールドを持たせるか
  - 自動検出か明示指定か
  - デフォルト言語は何か
  - ハイライトなし（plaintext）オプションは必要か
- **影響**: シンタックスハイライトの実装不可
- **議論**: 明示指定 vs 自動検出 → 明示指定で合意
- **決定**: ✅ ステージデータに`language`フィールド（必須、明示指定）。有効値: `"css"`,`"html"`,`"javascript"`,`"json"`,`"markdown"`,`"python"`,`"plaintext"`。自動検出なし。`plaintext`=ハイライトなし。B6のステージJSONスキーマに既に含まれている形を正式仕様とする。

## 🟡 DEFERRED → 全件確定（10件）

### D1. モード背景色の具体値
- **議論**: Normal=ベース色と同一 vs わずかに青寄り → ベース色と同一で合意
- **決定**: ✅ Normal=`#1a1a2e`（ベース色）、Insert=`#2e1a1a`（赤寄り暗色）、Visual=`#2a1a3e`

### D2. アニメーション詳細一覧
- **議論**: 未定義3項目+イージング関数を一括決定
- **決定**: ✅ 不正コマンド赤フラッシュ=0.15秒、トースト=2.0秒、lockedシェイク=0.3秒（左右2往復）。全共通ease-out、カード飛行のみease-in-out

### D3. キーボードレイアウト対応
- **議論**: `event.key`（論理キー）vs `event.code`（物理キー位置）→ Vimは論理キーベースなので`event.key`
- **決定**: ✅ `event.key`で判定。OSキーマップ準拠。Dvorak等でも特別対応不要、注記不要

### D4. テキストオブジェクトの境界ルール
- **議論**: Vim準拠3文字クラス分類で合意
- **決定**: ✅ keyword(`a-z A-Z 0-9 _`)/punctuation(非keyword非空白)/whitespace。`iw`on`hello-world`の`h`→`"hello"`。WORD=空白区切りのみ

### D5. `.`（ドットリピート）の非repeatable対応
- **議論**: テキスト変更操作のみ繰り返し可能で合意
- **決定**: ✅ 編集コマンド+Insert開始+入力=繰り返し可。移動/undo/検索/Visual開始/`.`自体=不可。変更なし時は何もしない、ダメージなし

### D6. レジスタのステージ間持ち越し
- **議論**: ステージ独立性を保つため全クリアで合意
- **決定**: ✅ ステージ開始時にunnamed/`"0`/`"a`〜`"z`全レジスタクリア

### D7. ヒントオーバーレイの具体的表示方法
- **議論**: 半透明オーバーレイ+初期状態デモで合意
- **決定**: ✅ `rgba(0,0,0,0.7)`オーバーレイ。初期状態からoptimalSpellを0.8秒間隔で自動実行。コマンド名を下部表示。任意キー/クリック/デモ完了1秒後に閉じる。再生中は入力ブロック

### D8. 範囲外コマンドのダメージ
- **議論**: シンプルに「キーを押したら1消費」で合意
- **決定**: ✅ コマンド発行=1ダメージ固定。移動なし/効果なしでも消費。手札外コマンドのみ0ダメージ

### D9. ブラウザ最低バージョン
- **議論**: 5年以上前のブラウザで十分カバーで合意
- **決定**: ✅ Chrome90+/Firefox88+/Safari14+/Edge90+。IE11非対応。非対応ブラウザはLandingで案内

### D10. ステージ設計ガイドライン
- **議論**: 概要ガイドラインのみ確定、詳細は後日で合意
- **決定**: ✅ 差分1-5箇所、☆3解法3-8コマンド、ライフ=☆1+余裕2-3、行数=序盤3-8/中盤8-20/終盤20-40、プレイ30秒-3分。詳細は別途STAGE-DESIGN-GUIDE.md
