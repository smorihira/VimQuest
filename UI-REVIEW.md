# VimQuest UI レビュー & 議論

モック全6画面を確認した上での問題点・議論事項を優先度順に整理。

---

## 🔴 Priority 1: 画面横断の構造問題

### 1.1 ナビゲーション・画面遷移フローの統一
- **現状の問題**: 各画面のヘッダー/戻る導線がバラバラ
  - スキルツリー: ヘッダーあり（進捗表示）
  - プレイ画面: ヘッダーなし
  - リザルト: ボタンのみ（もう一度 / 次のステージへ）
  - ゲームオーバー: ボタンのみ（ツリーに戻る / リトライ）
  - 武器習得: ボタンなし（キー入力で進む）
  - ランディング: ヘッダーなし
- **議論ポイント**:
  - 全画面に共通ヘッダーを入れるか？
  - 画面遷移フロー: Landing → Tree → Play → Result/GameOver → (Weapon) → Tree の流れをどう表現するか
  - プレイ中に「ツリーに戻る」「ギブアップ」導線は必要か
- **決定**: ✅ B案ベース採用
  - プレイ画面: ヘッダーなし（没入感優先）
  - Result/GameOver/Weapon: オーバーレイ的扱い、ヘッダーなし
  - Skill Tree: ヘッダーあり（ロゴ+進捗表示、ホーム画面的位置付け）
  - Landing: ヘッダーなし（1画面完結）
  - プレイ中のギブアップ: 左上に控えめアイコン + `:q!` 入力でツリーに戻れるイースターエッグ
  - Result: 「もう一度」「次のステージへ」「ツリーへ」の3択
  - GameOver: 「リトライ」「ツリーに戻る」の2択
  - Weapon: キー入力で自動遷移

### 1.2 画面遷移フロー図
- **現状の問題**: 仕様書に画面遷移の全体図がない
- **議論ポイント**: 遷移パターンの洗い出し
  - Landing → Tree（はじめる/つづける）
  - Tree → Play（ステージ選択）
  - Play → Result（クリア時）
  - Play → GameOver（ライフ0）
  - Result → Weapon（新武器アンロック時のみ）
  - Result → Tree / Result → Play（次 or リトライ）
  - GameOver → Play（リトライ）/ GameOver → Tree
  - Weapon → Tree / Weapon → Play（次ステージ）
- **決定**: ✅ 以下の遷移フローで確定
  - Landing → Tree（はじめる/つづける）
  - Tree → Play（ステージ選択）
  - Play → Result（クリア時）
  - Play → GameOver（ライフ0）
  - Result → Play（次ステージ直接遷移OK）/ Result → Tree
  - Result → Weapon（新武器アンロック時のみ）→ Tree（ツリーで開放状態を見せる）
  - GameOver → Play（リトライ）/ GameOver → Tree
  - Landingの再訪問導線は不要。Treeがホーム。Landingは初回 or URL直アクセス時のみ

## 🟡 Priority 2: 画面別の設計問題

### 2.1 Result画面: コマンド数の齟齬
- **現状の問題**: COMMANDS: 5 だが YOUR SPELL に4要素しか表示されていない
- **議論ポイント**:
  - Esc はコマンドカウントに含めるか？（仕様上 Esc はダメージ0）
  - テキスト入力 `#333333` はスペルとして1つにカウントするか、文字数ベースか
  - そもそも COMMANDS 数 = ダメージ数 ではない（Insert入力分の差）ため表示ルールの明確化
- **決定**: ✅ ラベルを `COMMANDS` → `DAMAGE` に変更。YOUR SPELLの各要素下に小さくダメージ数（-1, -0等）を表示。LIFE（残り/最大）も併記。

### 2.2 Result画面: 効率コマンドの強調ルール
- **現状の問題**: `ci"` だけ緑背景で強調されているが、ルールが不明
- **議論ポイント**:
  - 「そのステージで求められるキーコマンド」を強調するのか
  - 全コマンド同色でシンプルにするのか
  - 色分けのルール（新武器=強調、移動=薄め、など）
- **決定**: ✅ ステージの学習対象コマンドを緑（#44ddaa）で強調。それ以外は通常accent色。Esc/テキスト入力はdim色。

### 2.3 Result画面: 星表示の混乱
- **現状の問題**: 今回の星が ★★☆、BESTが ★★★ と小さく表示。視認性・意味の区別が弱い
- **議論ポイント**:
  - BEST表示は必要か？（モチベーション的にリトライ意欲を掻き立てる？）
  - 今回とBESTの見せ方をどう差別化するか
- **決定**: ✅ 大きな星は今回の結果のみ。BESTは星ではなく `BEST: 3 damage` のテキスト表示。BEST更新時は `NEW BEST!` を金色で追加。

### 2.4 Skill Tree: レイアウトの偏り
- **現状の問題**: ノードが左上に集中、右半分・下半分が空白
- **議論ポイント**:
  - 横スクロール前提でOKか？
  - 自動フィット（ズーム調整）するか
  - ツリーが大きくなったとき（v1.0の60コマンド全配置時）のレイアウト戦略
  - ミニマップの必要性
- **決定**: ✅ 初期表示は全ノード自動フィット。マウスドラッグでパン、ホイールでズーム。ミニマップはv1.0不要。availableノードにフォーカスするボタン [🎯] を用意。レイアウトは左→右、ブランチは下に伸びる構造を維持。

### 2.5 Skill Tree: locked ノードの視認性
- **現状の問題**: ロック状態のノードが薄すぎて読みにくい
- **議論ポイント**:
  - ロックノードのコマンド名は見せるべきか？（「次に何があるか」の期待感 vs 発見のサプライズ）
  - 見せる場合、どの程度の濃さにするか
- **決定**: ✅ コマンド名は見せる。opacity 0.35程度で「読めるが薄い」状態に。渇望感の演出として重要。

### 2.6 Game Over: ログの畳み込み
- **現状の問題**: `j` が3行連続で並ぶ → 冗長
- **議論ポイント**:
  - 繰り返しコマンドは `j ×3 (-3)` のように畳むか
  - 畳むと「浪費の具体性」が薄れるリスク
  - waste操作だけ展開、OK操作は畳む折衷案は？
- **決定**: ✅ 連続する同一コマンドは種類問わず畳む（`j ×3 -3`）。waste操作は赤ハイライト維持。一貫性のある表示。

### 2.7 Landing: Vim未経験者への伝達
- **現状の問題**: `ci"` デモは Vim を知らない人には意味不明の可能性
- **議論ポイント**:
  - ターゲットユーザーはVim初心者（ある程度知っている）か完全未経験者か
  - アニメーション付きデモ（ステップ実行）にするか
  - テキスト説明を添えるか
- **決定**: ✅ ターゲットはVim初心者と割り切り。デモ下に `ci" = "" の中身を丸ごと変更` 程度の短いキャプション1行を追加。ステップアニメーションはv1.0不要。

## � Round 2: モック修正後の追加問題点（2026-05-23）

### R2.1 Result: ライフ表示の矛盾・重複
- **現状の問題**:
  - 星の下に「残りライフ 5 / 10」と表示
  - Stats欄に「LIFE 7 / 10」と表示
  - 2箇所で数値が矛盾している（5と7）
  - そもそもライフ情報が2箇所あるのが冗長
- **議論ポイント**:
  - 星の下の「残りライフ」を削除し、Stats欄のLIFEに一本化するか
  - 逆に星の下のみに残し、Stats欄からLIFEを削除するか
  - 星の下には別の情報（ダメージ閾値など）を表示するか
- **決定**: ✅ 星の下のライフ表示を削除。Stats欄（DAMAGE / LIFE / BEST）に一本化。星は感情、Statsは分析と役割分離。

### R2.2 Result: NEW WEAPON と遷移の整合性
- **現状の問題**: NEW WEAPONバナーが表示されているが、「次のステージへ→」を押すと遷移フロー上はWeapon画面を経由するはず。ボタンラベルがそれを示唆していない
- **議論ポイント**:
  - NEW WEAPONがある場合、「次のステージへ」を「武器を受け取る→」に変えるか
  - NEW WEAPONバナー自体をクリッカブルにするか
  - ボタンラベルは変えず、内部的にWeapon画面を挟むだけでよいか
- **決定**: ✅ ボタンラベルは「次のステージへ→」のまま。NEW WEAPON時は内部的にWeapon画面を挟む。バナーはResultに残し期待感を演出。

### R2.3 Landing: 「はじめる」vs「つづける」
- **現状の問題**: LocalStorageに進捗がある場合でも「はじめる」のみ表示
- **議論ポイント**:
  - 進捗がある場合は「つづける」に変えるか
  - 「はじめる」「つづける」の2ボタンにするか
  - 「はじめる」1つで統一し、ツリー画面で進捗を判断させるか
- **決定**: ✅ ボタンは1つ。進捗なし→「はじめる」、進捗あり→「つづける」に自動変更。Landing自体はスキップしない。

### R2.4 Tree: 🎯 フォーカスボタン未実装
- **現状の問題**: 決定済みの「availableノードにフォーカスするボタン」がモックに未反映
- **議論ポイント**: ボタンの配置位置（ヘッダー右？画面右下フローティング？）
- **決定**: ✅ 右下フローティング丸ボタン（Google Maps現在地ボタンと同パターン）。押すとavailableノードにスムーズパン＋ズーム。

### R2.5 Play: :q! ボタンの確認
- **現状の問題**: モックに追加済みだが、スクリーンショットでの視認性が未確認
- **決定**: ✅ 確認済み。左上に `:q!` ボタンが控えめに表示されている

### R2.6 GameOver: 合計ダメージの不在
- **現状の問題**: ログで各操作のコストは見えるが、合計ダメージ / ライフ上限の情報がない。「あと何手節約すればクリアできたか」が分からない
- **議論ポイント**:
  - ログ下にサマリー行（`合計ダメージ: 14 / ライフ: 10 → 4オーバー`）を追加するか
  - HINTの「計3ダメージ」と対比できるようにするか
- **決定**: ✅ ログ下にサマリー行を追加（`合計ダメージ: 14 / ライフ: 10 → 4オーバー`）。HINTの最適解ダメージと対比して改善幅を可視化。

### R2.7 Play: エディタ領域とカード領域のバランス
- **現状の問題**: エディタが画面の大半を占め、スキルカード（HAND）が下部に押し込まれている。カードが小さく、ラベルが読みにくい
- **議論ポイント**:
  - カード領域の高さを広げるか
  - カードサイズを大きくするか、数が増えたときのスクロール/折り返し戦略
  - エディタとカードの比率（現状約 80:20）
- **決定**: ✅ エディタ:カード = 70:30 固定比率。カード領域を広げてカードサイズ・ラベル可読性を改善。

### R2.8 Play: カードの情報密度
- **現状の問題**: カードに `d` `c` `y` `f` `t` `iw` `i"` `a(` `ci"` が並んでいるが、カード下部のラベル（delete, change, yank等）が非常に小さい。合成系カード（`ci"`）と単体カード（`d`）の区別がカード色（青=合成）のみ
- **議論ポイント**:
  - ラベルをもっと読みやすくするか、ホバーで表示にするか
  - カードの枚数が多くなったとき（60コマンド中の手札）のUI戦略
  - 基本コマンド昇格（hjkl等は表示しない）のルールで手札枚数は適切か
- **決定**: ✅ カードにはコマンド名のみ大きく表示、ラベルはホバーツールチップ。手札は最大12〜15枚想定。2行表示可（上段=動詞・モーション、下段=テキストオブジェクト・合成）。

### R2.9 Play: ステージタイトルの視認性
- **現状の問題**: `STAGE 12 — ci" で属性値を書き換えろ` がエディタの上に表示されているが、NORMAL表示と縦に重なっている
- **議論ポイント**:
  - ステージタイトルとモード表示の配置を整理するか
  - タイトルは常時表示か、ステージ開始時のみ表示か
- **決定**: ✅ ステージタイトルとモード表示の配置を整理。実装時にTop bar内でレイアウト調整。
